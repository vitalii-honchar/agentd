# Data Model: Immutable Agent Revisions

## AgentRevision

Represents one immutable applied runtime artifact for an agent.

Fields:
- `agent_name`: Existing agent name.
- `revision_id`: UUID generated by the daemon at apply time.
- `content_digest`: Digest of normalized prompt, tools, environment material,
  copied file checksums, vendor, schedule, and runtime-relevant metadata.
- `source_path`: Definition markdown path used at apply time for audit only.
- `artifact_path`: Directory path for copied artifact files.
- `environment_json`: Captured environment entries from
  `environment.variables`, `environment.files`, and tool-specific env metadata,
  stored for execution and masked for inspection.
- `prompt`: Captured prompt text used by this revision.
- `vendor_name`, `vendor_model`: Captured model selection.
- `schedule_type`, `schedule_expression`: Captured schedule.
- `status`: `pending`, `finalized`, or `corrupt`.
- `created_at`: Time revision creation started.
- `finalized_at`: Time artifact validation completed.
- `error_message`: Corruption or creation failure detail when status is not finalized.

Validation rules:
- `(agent_name, revision_id)` is unique.
- `revision_id` must be a UUID.
- `artifact_path` must resolve under daemon-owned `data/work/<agent-name>/`.
- `finalized` revisions are immutable.
- Only finalized revisions can be executed.
- Environment file source paths must resolve during apply and must not be read
  from the source folder during execution.

State transitions:
- `pending` -> `finalized` after copy, manifest write, and validation.
- `pending` -> `corrupt` when recovery finds an incomplete artifact.
- `finalized` -> `corrupt` only when validation detects missing files before run.

## RevisionTool

Represents one frozen tool declaration attached to a revision.

Fields:
- `agent_name`
- `revision_id`
- `name`
- `kind`: `custom_tool`, `host_tool`, or `mcp_server`.
- `original_command`
- `rewritten_command`
- `args_json`
- `env_json`
- `timeout`
- `read_paths_json`
- `write_paths_json`
- `network_allow_json`
- `copied_files_json`

Validation rules:
- `custom_tool` implementations are copied into the artifact and use
  `rewritten_command` under the revision artifact directory.
- `host_tool` executables are not copied and keep the original host command or
  explicitly allowed absolute host path.
- `local_tool` is a legacy compatibility alias for `custom_tool` during
  migration and must not be emitted by new docs, examples, or API responses.
- Env values are available to execution but masked in default inspection output.
- Tool-level env values override global revision env values for that tool only.

## ToolExecutionLog

Represents observable evidence for one tool process in both AI Agent logs and
agentdserver logs.

Fields:
- `agent_name`
- `run_id`
- `revision_id`
- `tool_name`
- `tool_kind`
- `event_type`: `tool.execute.start`, `tool.execute.complete`, or
  `tool.execute.fail`.
- `stdout_summary`
- `stderr_summary`
- `result_summary`
- `exit_code`
- `timed_out`
- `error_message`
- `created_at`

Validation rules:
- AI Agent logs and agentdserver logs must include the same stdout, stderr,
  result, and error evidence for the tool execution.
- Raw environment values remain masked even when stdout or stderr summaries are
  logged.

## RevisionArtifactFile

Represents a copied runtime file in the artifact manifest.

Fields:
- `artifact_relative_path`
- `source_path`
- `sha256`
- `mode`
- `size_bytes`
- `copied_at`

Validation rules:
- `artifact_relative_path` must not escape the artifact directory.
- Executable mode is preserved for script files where supported.
- Symlinks are resolved or rejected according to the artifact copy policy; the
  finalized manifest records the chosen behavior.

## RevisionEnvironment

Represents environment material captured into a revision.

Fields:
- `agent_name`
- `revision_id`
- `key`
- `value`
- `source`: `literal`, `env_file`, or `tool_env`
- `source_path`: Original `.env` file path when source is `env_file`.
- `artifact_relative_path`: Copied `.env` file path when source is `env_file`.
- `masked`: Boolean used by inspection output.

Validation rules:
- Keys must be valid process environment names.
- Values are available to execution but masked in logs and default inspection.
- Literal `environment.variables` entries override keys parsed from
  `environment.files`.
- Tool-specific env entries override global revision env entries for that tool.

## ExecutionWorkDir

Represents the per-run process working directory.

Fields:
- `agent_name`
- `execution_id`: Run ID used for directory naming.
- `path`: `data/work/<agent_name>/executions/<execution_id>`.
- `revision_id`: Revision used by the run.
- `created_at`

Validation rules:
- The path must resolve under daemon-owned `data/work/<agent_name>/executions/`.
- Relative write paths resolve under this directory.
- The directory is mutable run output and must not mutate finalized revision
  artifact files.

## Agent

Existing applied agent record remains the latest control-plane view for one
agent name.

New relationships:
- Has many `AgentRevision` records.
- Latest active revision points to the newest finalized revision.

## AgentRun

Existing run record already stores `agent_revision`.

New validation:
- `agent_revision` must reference a finalized `AgentRevision` at run creation.
- Runs created with `<agent-name>:<revision>` preserve the explicit revision ID.
- Every run has a working directory under
  `data/work/<agent_name>/executions/<execution_id>`.
- Runtime reads immutable inputs from the revision artifact and writes generated
  files to the execution working directory.
